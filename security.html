1. Microsoft Identity Platform

Microsoft Identity Platform は、OAuth 2.0 / OpenID Connect に準拠した認証・認可プラットフォームです。

アプリ登録

Azure ポータルで「アプリの登録」を行い、クライアントID・テナントID・リダイレクトURIを設定します。

  • クライアントID (Application ID): アプリの一意識別子
  • テナントID (Directory ID): Azure AD テナントの識別子
  • リダイレクトURI: 認証後のリダイレクト先
  • クライアントシークレット / 証明書: サーバーアプリの認証に使用

OAuth 2.0 フロー

フロー用途トークン
Authorization CodeWeb アプリ（サーバーサイド）ID + Access + Refresh
Authorization Code + PKCESPA / モバイルアプリID + Access + Refresh
Client Credentialsデーモン / バックエンドサービスAccess のみ
Device Code入力制約のあるデバイスID + Access + Refresh
試験ポイント Client Credentials フローはユーザーの操作なしにアプリケーション自体として認証します。Delegated（委任）権限は使えず、Application 権限のみ使用できます。

MSAL（Microsoft Authentication Library）

C#
// パブリッククライアント（デスクトップ/モバイル）
var app = PublicClientApplicationBuilder
    .Create("clientId")
    .WithAuthority(AzureCloudInstance.AzurePublic, "tenantId")
    .WithRedirectUri("http://localhost")
    .Build();

var result = await app.AcquireTokenInteractive(scopes)
    .ExecuteAsync();
string token = result.AccessToken;

// コンフィデンシャルクライアント（Web アプリ / デーモン）
var app = ConfidentialClientApplicationBuilder
    .Create("clientId")
    .WithClientSecret("clientSecret")
    .WithAuthority(new Uri("https://login.microsoftonline.com/tenantId"))
    .Build();

// Client Credentials フロー
var result = await app.AcquireTokenForClient(scopes)
    .ExecuteAsync();
MSAL のクライアント種別 PublicClientApplication: ユーザーがいるクライアント（デスクトップ、モバイル、SPA）。 ConfidentialClientApplication: シークレットを安全に保持できるサーバーアプリ（Web アプリ、デーモン）。

トークンの種類

  • ID トークン: ユーザーの認証情報（JWT形式、クレーム含む）
  • アクセストークン: API へのアクセス許可（スコープ/権限情報含む）
  • リフレッシュトークン: 新しいアクセストークンを取得するために使用

2. Microsoft Graph

Microsoft Graph は、Microsoft 365 のデータにアクセスするための統一 API です。

エンドポイント

  • ベースURL: https://graph.microsoft.com/v1.0（安定版）, /beta（プレビュー）
  • ユーザー情報: GET /me
  • メール: GET /me/messages
  • ファイル: GET /me/drive/root/children
C#
// Graph SDK の使用
var credential = new ClientSecretCredential(
    "tenantId", "clientId", "clientSecret");
var graphClient = new GraphServiceClient(credential);

// ユーザー情報の取得
var user = await graphClient.Me.GetAsync();
Console.WriteLine(user.DisplayName);

3. Azure Key Vault

Key Vault は、シークレット、暗号化キー、SSL証明書を安全に管理するサービスです。

管理対象

種類説明例
シークレット文字列値（パスワード、接続文字列等）DB パスワード、API キー
キー暗号化キー（RSA、EC）データ暗号化、署名
証明書X.509 証明書の管理と自動更新SSL/TLS 証明書

アクセス制御

方式推奨度特徴
Azure RBAC推奨管理プレーンとデータプレーンの両方を制御。きめ細かい権限設定
アクセスポリシーレガシーVault レベルの権限設定。個別操作の制御が限定的

.NET SDK での操作

C#
// SecretClient の作成（マネージドIDを使用）
var client = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential());

// シークレットの取得
KeyVaultSecret secret = await client.GetSecretAsync("MySecret");
string value = secret.Value;

// シークレットの設定
await client.SetSecretAsync("MySecret", "newSecretValue");

// シークレットの削除（ソフトデリート）
DeleteSecretOperation operation = await client.StartDeleteSecretAsync("MySecret");
Azure CLI
# Key Vault の作成
az keyvault create --name myVault --resource-group myRG --location japaneast

# シークレットの設定
az keyvault secret set --vault-name myVault --name MySecret --value "secretValue"

# シークレットの取得
az keyvault secret show --vault-name myVault --name MySecret

4. マネージドID（Managed Identity）

マネージドID は、Azure サービスが他の Azure リソースに安全にアクセスするための機能です。資格情報の管理が不要です。

種類ライフサイクル共有ユースケース
システム割り当てリソースと同じ（削除で消滅）不可（1対1）単一リソース用
ユーザー割り当て独立（個別に管理）可（複数リソースで共有）複数リソースで同一権限

DefaultAzureCredential

DefaultAzureCredential は、複数の認証方法を自動的に試行するクラスです。

認証チェーンの順序 環境変数 → Workload Identity → Managed Identity → Azure CLI → Azure PowerShell → Interactive Browser の順に試行されます。ローカル開発では Azure CLI、本番環境ではマネージドIDが自動的に使われます。
C#
// DefaultAzureCredential で Key Vault にアクセス
var credential = new DefaultAzureCredential();

// Key Vault
var secretClient = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"), credential);

// Blob Storage
var blobClient = new BlobServiceClient(
    new Uri("https://mystorage.blob.core.windows.net/"), credential);
Azure CLI
# システム割り当てマネージドIDの有効化
az webapp identity assign --name myApp --resource-group myRG

# Key Vault へのアクセス許可（RBAC）
az role assignment create \
  --role "Key Vault Secrets User" \
  --assignee <principalId> \
  --scope /subscriptions/<subId>/resourceGroups/myRG/providers/Microsoft.KeyVault/vaults/myVault

5. Azure App Configuration

App Configuration は、アプリケーション設定と機能フラグを一元管理するサービスです。

主な機能

  • キー・値の管理: 名前空間、ラベルによる環境別設定
  • 機能フラグ: 機能の有効/無効をコード変更なしに切り替え
  • Key Vault 参照: シークレット値は Key Vault への参照として保存
  • スナップショット: 設定のポイントインタイム スナップショット
C#
// App Configuration クライアント
var client = new ConfigurationClient("connectionString");

// 設定値の取得
ConfigurationSetting setting = client.GetConfigurationSetting("MyApp:Settings:Message");
Console.WriteLine(setting.Value);

// 設定値の設定
client.SetConfigurationSetting("MyApp:Settings:Message", "Hello!");

// ラベル付き設定（環境別）
client.SetConfigurationSetting(new ConfigurationSetting(
    "MyApp:ConnectionString", "value", label: "Production"));

機能フラグ（Feature Flags）

C#
// ASP.NET Core での機能フラグ使用
// Program.cs
builder.Configuration.AddAzureAppConfiguration(options =>
{
    options.Connect("connectionString")
           .UseFeatureFlags();
});
builder.Services.AddFeatureManagement();

// Controller での使用
[FeatureGate("BetaFeature")]
public IActionResult BetaEndpoint()
{
    return Ok("Beta feature is enabled!");
}

📝 確認テスト

Q1

ユーザーの操作なしに、バックエンドサービスがAPIにアクセスするための OAuth 2.0 フローはどれですか？

Client Credentials フローはユーザーの介入なしにアプリケーション自体として認証します。デーモンやバックエンドサービスに最適です。
Q2

MSAL でサーバーサイド Web アプリケーションを構築する場合、使用するクラスはどれですか？

ConfidentialClientApplication は、クライアントシークレットや証明書を安全に保持できるサーバーサイドアプリで使用します。PublicClientApplication はデスクトップやモバイル向けです。
Q3

Key Vault のアクセス制御方式として、Microsoft が推奨しているのはどれですか？

Azure RBAC が推奨です。管理プレーンとデータプレーンの両方をきめ細かく制御でき、条件付きアクセスなどの高度な機能も利用可能です。
Q4

マネージドIDで、複数のリソース間で共有できるのはどちらですか？

ユーザー割り当てマネージドIDは、独立したリソースとして作成され、複数のAzureリソースで共有できます。システム割り当ては1つのリソースに紐づきます。
Q5

DefaultAzureCredential がローカル開発環境で最初に成功する可能性が高い認証方法は？

ローカル開発環境では通常、Azure CLI（az login）で認証済みのため、Azure CLI 認証が成功します。マネージドIDはAzure環境でのみ動作します。
Q6

Azure App Configuration でシークレット値を安全に保存する方法はどれですか？

App Configuration では、シークレット値を直接保存せず、Key Vault への参照として保存するのがベストプラクティスです。これにより、シークレットの一元管理とローテーションが可能になります。
Q7

Microsoft Graph API の安定版エンドポイントのベースURLはどれですか？

Microsoft Graph の安定版（v1.0）エンドポイントは https://graph.microsoft.com/v1.0 です。/beta はプレビュー版で、変更される可能性があります。
Q8

SPA（シングルページアプリケーション）で推奨される OAuth 2.0 フローはどれですか？

SPA では Authorization Code + PKCE（Proof Key for Code Exchange）が推奨されます。Implicit フローは非推奨です。PKCE はクライアントシークレットなしでセキュアな認証を実現します。
Q9

Key Vault でシークレットを削除したとき、デフォルトの動作はどれですか？

Key Vault ではソフトデリートがデフォルトで有効です。削除されたシークレットは保持期間（7-90日）中に復元可能です。
Q10

App Configuration で環境ごとに異なる設定値を管理する機能はどれですか？

ラベルを使って同じキーに対して環境ごとの値を設定できます（例: label="Development", label="Production"）。
