⚙ Azure コンピューティングソリューションの開発

出題比率: 25-30% ― 試験で最も重要な領域

1. Azure App Service Web Apps

App Service は、Web アプリ、REST API、モバイルバックエンドをホスティングするフルマネージドプラットフォームです。

App Service プラン

App Service プランは、アプリが動作するコンピューティングリソースを定義します。

レベルプラン特徴スケール
共有Free / Shared開発・テスト用。カスタムドメイン不可(Free)スケールアウト不可
専用Basic開発・テスト向け。手動スケール最大3インスタンス
専用Standard本番ワークロード。自動スケール・スロット最大10インスタンス
専用Premium v3高パフォーマンス・VNet統合最大30インスタンス
分離Isolated v2専用環境(ASE)・ネットワーク分離最大100インスタンス
試験ポイント デプロイメントスロットは Standard 以上のプランで利用可能です。Free/Shared/Basic では使えません。

Web アプリの作成（Azure CLI）

Azure CLI
# リソースグループの作成
az group create --name myResourceGroup --location japaneast

# App Service プランの作成
az appservice plan create \
  --name myAppServicePlan \
  --resource-group myResourceGroup \
  --sku S1 --is-linux

# Web アプリの作成
az webapp create \
  --resource-group myResourceGroup \
  --plan myAppServicePlan \
  --name myUniqueAppName \
  --runtime "DOTNETCORE:8.0"

アプリケーション設定と接続文字列

App Service では、アプリケーション設定は環境変数としてアプリに渡されます。

  • アプリ設定は OS の環境変数として公開される
  • 接続文字列も環境変数として公開される（プレフィックス付き: SQLAZURECONNSTR_ 等）
  • 設定値は Azure Portal 上で暗号化されて保存される
  • スロットに固定（sticky）する設定も可能
Azure CLI
# アプリ設定の追加
az webapp config appsettings set \
  --resource-group myResourceGroup \
  --name myUniqueAppName \
  --settings MY_SETTING=value1 DB_HOST=mydb.postgres.database.azure.com

# スロット固定設定
az webapp config appsettings set \
  --resource-group myResourceGroup \
  --name myUniqueAppName \
  --slot-settings SLOT_SPECIFIC=true

デプロイメントスロット

デプロイメントスロットを使うと、ダウンタイムなしでアプリをデプロイできます。

  • スワップ: staging スロットと production スロットを入れ替え
  • 自動スワップ: デプロイ完了後に自動的にスワップ
  • トラフィックルーティング: 一定割合のトラフィックを別スロットに転送（A/Bテスト）
スワップ時の動作 スワップ時にリセットされる設定: 接続文字列のスロット固定設定、認証設定。 スワップ時にリセットされない設定: カスタムドメイン、SSL証明書、スケール設定。
Azure CLI
# スロットの作成
az webapp deployment slot create \
  --name myUniqueAppName \
  --resource-group myResourceGroup \
  --slot staging

# スロットのスワップ
az webapp deployment slot swap \
  --name myUniqueAppName \
  --resource-group myResourceGroup \
  --slot staging \
  --target-slot production

組み込み認証（Easy Auth）

App Service にはコード変更不要の認証機能が組み込まれています。

  • Microsoft Entra ID（Azure AD）
  • Facebook / Google / Twitter / GitHub
  • 認証トークンはリクエストヘッダーで渡される（X-MS-TOKEN-*）

2. Azure Functions

Azure Functions は、イベント駆動型のサーバーレスコンピューティングサービスです。

ホスティングプラン比較

プランスケール最大タイムアウトVNet料金
Consumptionイベント駆動で自動5分(既定)/10分(最大)不可実行回数+実行時間
Premiumイベント駆動で自動（事前ウォーム）無制限可常時ウォームインスタンス+追加
Dedicated (App Service)手動/自動スケール無制限可App Serviceプラン料金
試験ポイント Consumption プランのデフォルトタイムアウトは5分、最大10分。長時間処理にはPremiumまたはDedicatedプランを使用します。

トリガーとバインディング

トリガーは関数の起動方法を定義し、バインディングは入出力データの接続を宣言的に行います。

トリガー入力バインディング出力バインディング
HTTP-HTTP Response
Timer--
Blob StorageBlobBlob
Queue Storage-Queue
Cosmos DBCosmos DBCosmos DB
Event Grid-Event Grid
Service Bus-Service Bus

HTTP トリガー関数（C#）

C#
[Function("HelloWorld")]
public static IActionResult Run(
    [HttpTrigger(AuthorizationLevel.Function,
      "get", "post")] HttpRequest req,
    ILogger log)
{
    log.LogInformation("HTTP trigger function processed a request.");
    string name = req.Query["name"];
    return new OkObjectResult($"Hello, {name}");
}

Timer トリガー（CRON式）

C#
// 5分ごとに実行: "0 */5 * * * *"
// 毎日午前9時: "0 0 9 * * *"
// CRON形式: {秒} {分} {時} {日} {月} {曜日}
[Function("TimerJob")]
public static void Run(
    [TimerTrigger("0 */5 * * * *")] TimerInfo timer,
    ILogger log)
{
    log.LogInformation($"Timer executed at: {DateTime.Now}");
}

Durable Functions

Durable Functions は、ステートフルなワークフローをサーバーレスで実現する拡張機能です。

パターン説明ユースケース
Function Chaining関数を順番に実行注文処理パイプライン
Fan-out/Fan-in複数関数を並列実行→結果を集約バッチ処理
Async HTTP APIs外部からの長時間操作の状態確認レポート生成
Monitor定期的にポーリングジョブ完了待ち
Human Interaction人間の承認を待つ承認ワークフロー
C#
// オーケストレーター関数（Function Chaining）
[Function("ChainOrchestrator")]
public static async Task<string> RunOrchestrator(
    [OrchestrationTrigger] TaskOrchestrationContext context)
{
    var result1 = await context.CallActivityAsync<string>("Step1", null);
    var result2 = await context.CallActivityAsync<string>("Step2", result1);
    var result3 = await context.CallActivityAsync<string>("Step3", result2);
    return result3;
}

3. コンテナソリューション

Azure Container Registry (ACR)

ACR はプライベートな Docker コンテナレジストリです。

SKUストレージWebhookGeo レプリケーションACR Tasks
Basic10 GB2不可可
Standard100 GB10不可可
Premium500 GB500可可
Azure CLI
# ACR の作成
az acr create --resource-group myRG --name myRegistry --sku Standard

# Docker イメージのビルド（ACR Tasks）
az acr build --registry myRegistry --image myapp:v1 .

# ACR へのログイン
az acr login --name myRegistry

# イメージのプッシュ
docker tag myapp:v1 myRegistry.azurecr.io/myapp:v1
docker push myRegistry.azurecr.io/myapp:v1

Azure Container Instances (ACI)

ACI は、最も高速にコンテナを実行する方法です。VMの管理不要で、秒単位の課金です。

Azure CLI
# コンテナインスタンスの作成
az container create \
  --resource-group myRG \
  --name mycontainer \
  --image myRegistry.azurecr.io/myapp:v1 \
  --cpu 1 --memory 1.5 \
  --registry-login-server myRegistry.azurecr.io \
  --registry-username myUser \
  --registry-password myPass \
  --dns-name-label myapp-demo \
  --ports 80

# コンテナグループ（YAML定義）
az container create --resource-group myRG \
  --file deploy-aci.yaml
コンテナグループ ACI ではコンテナグループにより複数のコンテナを同一ホスト上で実行できます（サイドカーパターン）。コンテナグループ内のコンテナは、ライフサイクル、リソース、ローカルネットワーク、ストレージボリュームを共有します。

Azure Container Apps

Container Apps は、マイクロサービスやイベント駆動アプリに最適なサーバーレスコンテナプラットフォームです。

  • リビジョン: デプロイごとに不変のスナップショットが作成される
  • スケーリングルール: HTTP同時リクエスト数、Queue長、カスタムメトリクスに基づく
  • 0へのスケールインが可能（HTTPトリガー以外の場合）
  • Dapr統合: サービス間通信、状態管理、Pub/Sub
  • イングレス: 外部/内部HTTPトラフィックの管理
Azure CLI
# Container Apps 環境の作成
az containerapp env create \
  --name myEnvironment \
  --resource-group myRG \
  --location japaneast

# Container App の作成
az containerapp create \
  --name myapp \
  --resource-group myRG \
  --environment myEnvironment \
  --image myRegistry.azurecr.io/myapp:v1 \
  --target-port 80 \
  --ingress external \
  --min-replicas 0 \
  --max-replicas 10
ACI vs Container Apps ACI: 短時間のタスク、バッチ処理に最適。Container Apps: 長時間稼働するマイクロサービス、自動スケーリングが必要な場合に最適。

📝 確認テスト

Q1

App Service でデプロイメントスロットを使用するために必要な最小のプランはどれですか？

デプロイメントスロットは Standard (S1) 以上のプランで利用可能です。Free、Shared、Basic プランではスロット機能は使えません。
Q2

Azure Functions の Consumption プランにおけるデフォルトのタイムアウトは何分ですか？

Consumption プランのデフォルトタイムアウトは5分です。host.json の functionTimeout で最大10分まで延長可能です。
Q3

Durable Functions で、複数のアクティビティ関数を並列に実行して結果を集約するパターンはどれですか？

Fan-out/Fan-in パターンは、複数の関数を並列に実行し、全ての完了を待って結果を集約します。Function Chaining は逐次実行です。
Q4

Azure Container Registry で Geo レプリケーションをサポートする SKU はどれですか？

Geo レプリケーションは Premium SKU でのみ利用可能です。複数リージョンにレジストリを複製し、低レイテンシーのアクセスを実現します。
Q5

Azure Container Instances (ACI) の主な特徴として正しいものはどれですか？

ACI は VM の管理が不要で、秒単位で課金されます。自動スケーリングは組み込まれておらず（Container Apps がその役割）、Kubernetes も不要です。
Q6

App Service のスロットスワップ時に、ターゲットスロットに引き継がれない（リセットされる）ものはどれですか？

スロット固定（slot-specific）に設定されたアプリ設定や接続文字列は、スワップ時にそのスロットに残り、アプリと一緒には移動しません。カスタムドメインやSSL証明書、スケール設定はスワップに関係しません。
Q7

Azure Functions で、Cosmos DB のドキュメントが変更されたときに自動的に関数を実行するには、どのトリガーを使用しますか？

Cosmos DB トリガーは Change Feed を利用して、ドキュメントの作成・更新を検出し、自動的に関数を呼び出します。
Q8

Azure Container Apps で、HTTP リクエストがないときにレプリカを0にスケールインできますか？

Container Apps では min-replicas を 0 に設定可能ですが、HTTP スケールルールを使用する場合はデフォルトで最低1レプリカが必要です。キューやカスタムメトリクスベースのスケーリングでは 0 へのスケールインが可能です。
Q9

az acr build コマンドの特徴として正しいものはどれですか？

az acr build は ACR Tasks を使ってクラウド上でイメージをビルドします。ローカルに Docker をインストールする必要がないのが大きな利点です。全SKUで利用可能です。
Q10

Azure Functions で VNet 統合が必要な場合、どのホスティングプランを選択すべきですか？

VNet 統合は Premium プランと Dedicated (App Service) プランで利用可能です。Consumption プランでは VNet 統合はサポートされていません。
